# Implementation Plan: Guided UI для Telegram KB Bot

## Статус на 2026-03-30

### Что уже реализовано

1. `Phase 1` выполнен:
   - `/start` показывает главное меню.
   - Добавлен отдельный router для меню (`menu.py`).
   - Вынесены UI-клавиатуры и callback namespace в `bot/ui/`.
2. `Phase 2` выполнен в практичном объёме:
   - guided-flow добавления работает через меню.
   - выбор темы доступен кнопками.
3. `Phase 3` выполнен:
   - поиск через guided-flow.

   - пагинация списка/поиска (`◀ Назад` / `Далее ▶`).

4. `Phase 4` частично выполнен:
   - темы и коллекции доступны через кнопки и экраны.
5. `Phase 5` частично выполнен:
   - `Создать backup` и `Показать backups` через UI уже есть.
   - restore остаётся командным безопасным сценарием.

### Что подтверждено проверками

- локально: `python -m pytest -q` -> `76 passed`
- вручную:
  - возврат в главное меню из разных экранов.
  - пагинация в списках.

### Что осталось на следующую итерацию

1. UX-полировка текстов в экранах (особенно пустые выборки и пояснения по статусам).
2. Дополнительные интеграционные UI-тесты на сценарии callback-навигации.
3. При необходимости расширение paged-flow на дополнительные экраны.

## Цель первой итерации

В первой итерации не строим полноценный визуальный `Web App`, а добавляем удобный слой навигации внутри Telegram-бота:

- главное меню
- inline-кнопки
- пошаговые сценарии
- выбор сущностей без ручного ввода UUID

Командный режим остается рабочим.

## Ограничения и рамки

1. Полноценный `drag and drop` в обычном Telegram-чате не планируем.
2. В первую итерацию не ломаем существующие handlers и сервисы.
3. Новая UI-логика должна быть тонким слоем над уже существующими application services.
4. Все новые UX-сценарии должны поддерживать отмену и возврат назад.

## Phase 1: Главное меню и общая навигация

## Phase 2: Guided flow для добавления записи

### Что делаем

1. Сценарий `Добавить` запускается из меню.
2. Пользователь выбирает:
   - `URL`
   - `Заметка`
3. Затем бот по шагам собирает:
   - контент
   - заголовок
   - тему
4. Список тем показывается кнопками с пагинацией или батчами, а не через UUID-ввод.
5. После подтверждения бот показывает итоговую карточку созданной записи.

### Что переиспользуем

- FSM из текущего `/add`
- `EntryService`
- `TopicsRepository`
- `StatusesRepository`

### Что меняем

- `src/kb_bot/bot/handlers/add.py`
- `src/kb_bot/bot/fsm/states.py`

### Дополнительно

Текущую команду `/add` не удаляем. Либо она запускает тот же guided flow, либо остается отдельным legacy-entry point.

4. Если добавлять restore в UI, то только через двухшаговое подтверждение и отображение предупреждения.

### Что меняем

## Предлагаемая структура callback namespace

Примерно так:

- `menu:main`
- `menu:add`
- `menu:search`
- `menu:list`
- `entry:view:<id>`
- `entry:status:<id>:<status>`
- `topic:pick:<id>`
- `topic:page:<n>`
- `list:page:<n>`
- `search:page:<n>`
- `backup:list`
- `backup:create`

Нужен единый формат, чтобы callback-handlers не превратились в хаос.

4. Команды по-прежнему работают.
5. Тесты зеленые.
